## Decisões técnicas

- `convertToPost.ts` usa `TablesInsert<"posts">` em vez de `as never` para type safety
- Tabela `content_packages` antiga no banco tinha schema diferente — foi dropada e recriada (zero rows). Nunca usar `CREATE TABLE IF NOT EXISTS` ao mudar schema sem checar se a tabela existente é compatível
- Índice parcial `WHERE status not in (...)` falha com erro "column status does not exist" quando a tabela já existe com nome igual mas colunas diferentes — resolver com DROP + CREATE fresh
- `eslint-disable-line react-hooks/set-state-in-effect` nos `useEffect(() => { void loadX(); })` — padrão existente no projeto, não alterar
- `key={idea.id}` no EditDialogInner para reset de estado sem useEffect — padrão a manter em dialogs que recebem props mutáveis

## Problemas resolvidos

- **`content_packages` antiga com schema incompatível**: `CREATE TABLE IF NOT EXISTS` não recriou. Solução: checar com `SELECT column_name FROM information_schema.columns` antes de aplicar migration
- **`convertToPost.ts` build error**: `.from("posts" as never).insert(...)` quebrava tipo → reescrito com `TablesInsert<"posts">`
- **Deploy 401**: CLI Supabase sem sessão ativa — usar token de acesso via `SUPABASE_ACCESS_TOKEN=<token> supabase functions deploy ...`
- **ReelContent sem cenas**: prompt gera `scenes[]` mas componente só renderizava `hook_3s` e `cta_final` — corrigido com tipo e render completo

- Retry apenas em 503/429/529 e erros de rede — outros erros falham direto
- `agent-1-strategist` e `agent-2-roteirista` envolvem chamadas Gemini e Anthropic com `withRetry`

- `supabase db push` executado — migrations Sprint 1 + Sprint 2 aplicadas no remoto ✅
- `agent-2-roteirista` deployado com sucesso

### Decisões técnicas

- Helper em `_shared/` para reutilizar em ambas as functions sem duplicação
- `withRetry` nunca throwa — retorna `RetryResult<T>` e o caller decide
- `input_payload` enriquecido com `attempts` via update após `withRetry` (não bloqueia a chamada LLM)
- Merge de `feat/sprint2-agent-roteirista` na branch `fix/llm-retry-backoff` antes de aplicar o hotfix (main estava desatualizado)

### Problemas resolvidos

- **Gemini 503 em produção**: antes marcava run como `failed` imediatamente. Agora faz até 2 retries com backoff
- **ANTHROPIC_API_KEY vazia**: detectado que `.env.local` não tinha a key — Gemini voltou a funcionar antes de precisar do fallback
